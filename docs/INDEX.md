## Documentation Standards

### File Naming Conventions

- **Guides**: `{SYSTEM}_GUIDE.md` (e.g., `FIREBASE_GUIDE.md`)
- **Quick References**: `{SYSTEM}_QUICK_REFERENCE.md`
- **Summaries**: `{SYSTEM}_SUMMARY.md`
- **Reports**: `{AGENT/BATCH}_REPORT.md`
- **Data Files**: `{purpose}.json` (lowercase with hyphens)

### Documentation Types

1. **System Documentation** (`docs/systems/`)
   - Active guides and references
   - Implementation details
   - API documentation
   - Quick start guides

2. **Migration History** (`docs/migration/`)
   - Historical migration records
   - Batch processing logs
   - Upgrade documentation
   - Extraction data

3. **Validation Reports** (`docs/validation/`)
   - Test results
   - Quality assurance reports
   - Audit findings
   - Validation data

4. **Agent Reports** (`docs/agents/`)
   - Automated fix reports
   - Enhancement summaries
   - Processing logs

5. **Archive** (`docs/archive/`)
   - Deprecated documentation
   - Superseded guides
   - Historical reference

## Contributing to Documentation

### Adding New Documentation

1. Determine the correct category (systems/migration/validation/agents)
2. Follow naming conventions
3. Include frontmatter with date and description
4. Update this INDEX.md
5. Link from README.md if appropriate

### Updating Existing Documentation

1. Read the existing file
2. Make changes
3. Update "Last Updated" date
4. Update this INDEX.md if title/purpose changed

### Archiving Documentation

1. Move to `docs/archive/`
2. Add "DEPRECATED" notice to top of file
3. Update INDEX.md to reflect archive status
4. Update any links pointing to archived doc
